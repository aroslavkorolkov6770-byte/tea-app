# TEA HUB: контекст проекта

## Назначение

TEA HUB - закрытая B2B-платформа для обучения сотрудников чайной компании. В ней есть обучение, тесты и аттестации, база документов, ассортимент, каталог продуктов, профиль, уведомления, администрирование сотрудников и ИИ-помощник.

Главная страница доступна до входа. Рабочие разделы открываются после авторизации. Сайт работает на русском языке, в темной и светлой темах.

## Данные и критическое правило их сохранности

Сейчас проект использует JSON-хранилище. Ключевые файлы в `data/`:

- `tea_hub_users_v1.json` - учетные записи и роли.
- `tea_hub_products_v1.json` - товары.
- `tea_hub_assortment_matrix_v1.json` - дерево ассортимента.
- `tea_hub_dynamic_route_v2.json` и `tea_hub_dynamic_basics_v2.json` - обучение и тесты.
- `tea_hub_urgent_files_v1.json` и `file_data_*.json` - документы и их содержимое.
- `tea_hub_notifications_v1.json` - уведомления.
- `tea_hub_test_results_v1.json` - результаты тестов.
- `th_ai_history_*.json` - история отдельных ИИ-диалогов.
- `profile_data_*.json` - данные профилей.

Строго запрещено очищать, удалять, перезаписывать пустыми данными или массово заменять серверную папку `data/`. При разворачивании нужно обновлять код отдельно от пользовательских данных. Если есть сомнение, сначала создать резервную копию данных на сервере.

На финальном хосте планируется переход на MySQL. Это возможно, но требует отдельной миграции: сначала схемы и резервной копии, затем переноса данных, затем переключения API. Нельзя просто заменить JSON-файлы на MySQL без слоя миграции.

## Роли и доступ

- Администратор управляет сотрудниками, материалами, тестами, документами, ассортиментом и продуктами.
- Сотрудник видит свой обучающий путь и доступные материалы.
- Системный администратор - отдельный служебный профиль с отдельным идентификатором и отдельной историей ИИ. Его личные данные и профиль должны быть недоступны обычным администраторам. В админке он виден как служебная учетная запись, а его статистика скрыта.
- Тестовые учетные записи существуют для проверок. Не менять их логины, пароли, роли и данные без прямого указания пользователя.

Пароли должны храниться только в защищенном виде. В документации, коммитах и исходном коде не размещать действующие пароли, токены, ключи API, VAPID-ключи и SSH-ключи.

## ИИ-помощник

- Запросы из интерфейса идут в `/api/ai-chat`.
- В разделе обучения администратор может выбрать создание темы или теста через Alice AI. Генератор принимает до 8 файлов общим размером до 25 МБ, извлекает текст и формирует структурированный черновик через `/api/ai-content-draft`.
- Поддержаны PDF, DOCX, XLS/XLSX/ODS, PPTX, ODT/ODP, RTF и обычные текстовые форматы. Старые бинарные DOC/PPT, изображения без OCR, аудио и видео пока не извлекаются.
- AI-результат никогда не публикуется автоматически. Он открывается в существующем редакторе темы или теста, где администратор проверяет и исправляет его, а затем вручную сохраняет.
- Маршрут генератора доступен только роли `admin`, ограничивает размер файлов и прямо запрещает модели выполнять инструкции из загруженных документов.
- В прошлой работе исправлялся формат запросов внешнему ИИ-провайдеру и ошибка 400.
- История ИИ должна быть изолирована по ID пользователя. Особенно важно не объединить историю системного администратора с историей обычного администратора или сотрудника.
- При любых изменениях ИИ сначала проверить локально минимум два разных профиля и убедиться, что диалоги не смешиваются.

## Важные правила интерфейса

- Основной акцентный цвет: `#0abab5`.
- Для интерактивных кнопок применяется общая анимация наведения: легкое вдавливание, бирюзовая рамка и мягкая тень. Не делать новую, несовместимую анимацию без необходимости.
- В мобильной версии не должны появляться горизонтальные переполнения, обрезанные карточки и наложения текста на кнопки.
- Текст внутри карточек должен переноситься и не выходить за границы.
- Светлая тема должна заменять все темные поверхности на светлые, но фоновая фотография на главной странице должна оставаться исходной: без осветляющего фильтра и без белой подложки поверх изображения.

## Безопасная схема заливки

Заливка разрешена только после явной команды пользователя: «залей на гитхаб», «залей на хост» или «залей на гитхаб потом на хост».

Последовательность при команде «на GitHub, потом на хост»:

1. Проверить изменения: `git status`, `git diff`, затем убедиться, что файлы `data/` не попали в индекс.
2. Запустить проверку и сборку.
3. Добавить только файлы кода, создать короткий ясный коммит и выполнить `git push`.
4. Подключиться к заранее настроенному серверу по SSH, перейти в каталог приложения `/var/www/html`, выполнить `git pull --ff-only`. Если изменились зависимости, выполнить `npm install`, затем `npm run build` и `pm2 restart tea-hub`.
5. Не использовать команды очистки кэша, удаления данных, `git clean`, `git reset --hard` или копирование поверх всей папки приложения.
6. Проверить сайт по публичному адресу после перезапуска.

Точные адреса, пароли, токены и SSH-ключи не записываются в этот файл. Они должны находиться только в защищенной конфигурации пользователя и сервера.

### Команды деплоя

```powershell
# Локальная машина: только после проверки и осознанного коммита.
git add <только файлы кода>
git commit -m "Короткое понятное описание"
git push

# Сервер: после успешного push.
ssh <настроенное-подключение-к-серверу>
cd /var/www/html
git pull --ff-only
npm install # обязательно, если изменились package.json или package-lock.json
npm run build
pm2 restart tea-hub
```

## Известная проверка качества

`npm run lint` в текущем состоянии репозитория выдает большое количество старых ошибок типизации и правил React в файлах, не относящихся к текущим изменениям. Это накопленный технический долг, а не результат правок светлой темы. Для конкретной задачи ориентироваться на `npm run build`, просмотр страниц в браузере и точечную проверку измененных файлов.
