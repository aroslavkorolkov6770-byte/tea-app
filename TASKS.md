# TEA HUB: задачи и статус

## Как пользоваться этим списком

- `Готово` - работа уже выполнена и была зафиксирована в истории проекта.
- `В работе` - локальные изменения существуют, но еще не завершены проверкой и не залиты.
- `Ожидает` - задача не начата или требует отдельного решения.
- Любая внешняя заливка на GitHub или хост выполняется только по прямой команде пользователя.

### Создание тем и тестов через Alice AI

Статус: `Готово и развернуто`.

Локально готово:

1. Компактный выбор `Создать вручную` или `Создать через Alice AI` у тем и тестов.
2. Загрузка до 8 документов, не более 10 МБ на файл и 25 МБ суммарно.
3. Чтение PDF, DOCX, XLS/XLSX/ODS, PPTX, ODT/ODP, RTF и текстовых форматов.
4. Настройки раздела, детализации, сложности и количества вопросов.
5. Администраторский API с проверкой роли, лимитов и структуры ответа Alice AI.
6. Передача результата в существующий редактор без автоматической публикации.
7. Отдельные светлые и темные стили, адаптивная компоновка окна.
8. Успешные `npx tsc --noEmit` и `npm run build`; новые отдельные файлы проходят ESLint.

Проверка релиза:

1. Функциональный commit `8dedc82` отправлен в рабочую ветку и `main` на GitHub.
2. На хосте установлены новые зависимости, production build прошел, PM2-процесс перезапущен и работает.
3. `https://tea-hub.ru/` отвечает кодом `200`.
4. Публичный AI API принял Markdown-документ и вернул тему из двух блоков без предупреждений.
5. Меню и окно генератора проверены визуально на публичной странице обучения.
6. Тестовый AI-черновик не сохранялся в учебные данные.

Отдельными будущими задачами остаются OCR изображений, распознавание аудио и поддержка старых бинарных DOC/PPT.

## Важные постоянные требования

1. Не удалять и не очищать JSON-данные на сервере.
2. Не смешивать ИИ-историю разных пользователей.
3. Не ломать тестовые аккаунты и служебный системный аккаунт.
4. После импорта продуктов данные должны сохраняться после перезагрузки и перехода между разделами.
5. Если приложение аварийно теряет сессию, пользователь должен попадать на главную страницу, а не на обломок защищенного интерфейса.
6. Все новые элементы должны быть адаптивны и не давать тексту выходить за карточки.

## Ожидает отдельного планирования

### Производительность и надежность

- Ускорить загрузку обучения, тестов, документов, сотрудников и продуктов.
- Уменьшить зависания после импорта большого каталога продуктов.
- Добавить быстрый, безопасный клиентский кэш с серверной валидацией данных.
- Проверить накопление логов и размер файлов данных на сервере без их удаления.
- Устранить исчезновение данных документов и уроков, если проблема повторится после проверки API.

### Работа с материалами

- Удобное копирование документа в другой раздел.
- Изменение порядкового номера урока и теста с автоматическим сдвигом остальных карточек.
- Синхронизация ИИ с новой нумерацией.
- Унифицированный просмотр базовых форматов файлов без лишних транзакций.

### Продукты и ассортимент

- При появлении новой категории в импортированной таблице автоматически добавлять соответствующую ветку в ассортимент.
- При изменении порядка ассортимента отражать тот же порядок в продуктах.
- Контрольная проверка очистки всех продуктов с обязательным подтверждением.
- Экспорт без колонки даты добавления.

### Поиск, ИИ и уведомления

- Общий поиск должен видеть товары, категории и карточки продуктов.
- ИИ должен полноценно искать сведения по продуктам и всему разрешенному содержимому сайта.
- Ускорить отправку Web Push и e-mail уведомлений.

### Переход на MySQL

- Спроектировать таблицы, индексы, резервное копирование и миграцию JSON-данных.
- Перевести API на слой репозитория, чтобы фронтенд не зависел от формата хранилища.
- Провести миграцию только после резервной копии и теста на отдельной среде.
