# Ватэс: работа с GitHub и сервером

## Главное правило

Заливка выполняется только после прямой команды пользователя. Формулировка «залей на GitHub потом на хост» означает:

1. Сначала подготовить и отправить код в GitHub.
2. Затем обновить код на сервере.
3. Установить зависимости, если изменились `package.json` или `package-lock.json`.
4. Собрать приложение на сервере.
5. Перезапустить процесс приложения.
6. Проверить публичный сайт.

## Перед каждой заливкой

```powershell
git status --short
git diff --check
git diff --stat
```

Обязательно убедиться, что в коммит не входят:

- `data/*.json`;
- загруженные файлы пользователей;
- `.env*`;
- SSH-ключи;
- файлы с паролями, токенами, VAPID-ключами и API-ключами;
- случайные временные файлы.

## Коммит

Коммит должен быть коротким и понятным. Примеры:

- `Исправить светлую тему`
- `Улучшить навигацию по товарам`
- `Добавить сброс пароля сотрудника`

Нельзя писать расплывчатые сообщения вроде `fix`, `update`, `changes`.

## Обновление на сервере

После успешного `git push` подключиться к уже настроенному серверу по SSH и выполнить строго эту последовательность:

```bash
cd /var/www/html
git pull --ff-only
npm install
npm run build
pm2 restart tea-hub
```

`npm install` обязателен после изменения списка пакетов. Если зависимости в релизе не менялись, этот шаг можно пропустить.

`/var/www/html` - рабочий каталог сайта на текущем сервере. `tea-hub` - имя процесса в PM2. Нельзя удалять каталог приложения целиком, выполнять очистку данных, `git clean`, `git reset --hard`, копирование поверх `data/` или любые команды, которые могут удалить загруженные пользователями документы и JSON-хранилище.

## После обновления

- Открыть публичный сайт в браузере.
- Проверить главную страницу, вход, продукты, документы и ИИ-помощник.
- Убедиться, что данные не исчезли после перезагрузки страницы.
- При ошибке сначала сохранить текст ошибки и состояние процесса; не начинать очистку данных.

## Секреты

Секреты существуют только на защищенной машине или в защищенной конфигурации:

- SSH-ключи;
- доступ к серверу;
- ключи ИИ-провайдера;
- VAPID-ключи;
- почтовые учетные данные;
- секрет авторизации.

Не копировать их в исходный код, Markdown-файлы, коммиты, issue, PR и сообщения чата.
